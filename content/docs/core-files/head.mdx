A Zazz page is plain HTML. There's no app shell to configure and no build output to wire up — but a few things in the `<head>` and around your landmarks matter, because styles and scripts depend on them. Get these right once and everything else just works.

## The minimum head

This is all most pages need:

```html
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />

  <!-- Paint the right theme (background, form controls, scrollbars) before CSS loads -->
  <meta name="color-scheme" content="light dark" />

  <!-- One stylesheet — @imports every layer in cascade order -->
  <link rel="stylesheet" href="./zazz/styles/main.css" />

  <title>Page title</title>
</head>
```

### Declare your color scheme first

The `<meta name="color-scheme" content="light dark">` tag tells the browser you support both themes *before it starts rendering*. That lets it paint the canvas background, scrollbars, and form controls in the right mode immediately, instead of flashing white and correcting once your CSS arrives.

<Callout type="warn">
Keep the meta tag and the CSS `color-scheme` on `:root` in agreement — both declare `light dark`. The meta tag handles the pre-render paint; the CSS property drives [`light-dark()`](/docs/core-concepts/dark-mode) once styles load.
</Callout>

`color-scheme` is [Baseline](https://web.dev/baseline) and widely available (since 2022). Older browsers ignore the tag and render light mode — a safe default.

### Link one stylesheet

Point a single `<link>` at `main.css`. It `@import`s every Zazz file in the correct [cascade order](/docs/core-files/layers), so one link gives you every token, component, and utility:

```html
<link rel="stylesheet" href="./zazz/styles/main.css" />
```

<Callout>
A separate `<link rel="preload" as="style">` for `main.css` is **not** needed — a same-document stylesheet link is already the highest-priority, render-blocking fetch. `preload` is for *late-discovered* resources (web fonts, CSS pulled in by JS), not the stylesheet you link in `<head>`.
</Callout>

For production, keep the single link and turn on **brotli or gzip** at your host. CSS this regular compresses to roughly 10–15% of its raw size — a far bigger win than hand-splitting files, with nothing to keep in sync.

### Add your overrides after Zazz

Your own stylesheet loads *after* `main.css` so it can reassign any token. See [Adding custom styles](/docs/core-concepts/custom-styles) for what to put in it.

```html
<link rel="stylesheet" href="./zazz/styles/main.css" />
<link rel="stylesheet" href="./your-styles.css" />
```

### Loading a web font

Zazz defaults to **Geist** through `--font-family-body` and `--font-family-heading`. Point those tokens at any font you load. Because a web font is a late-discovered resource, this is the one place `preload` earns its keep:

```html
<link rel="preconnect" href="https://fonts.example.com" crossorigin />
<link rel="preload" as="font" type="font/woff2"
      href="https://fonts.example.com/your-font.woff2" crossorigin />
```

```css
/* your-styles.css */
:root {
  --font-family-body: "Your Font", var(--font-sans);
}
```

## The full document skeleton

A complete page wraps content in three landmarks and loads scripts at the end of `<body>`:

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <meta name="color-scheme" content="light dark" />
  <link rel="stylesheet" href="./zazz/styles/main.css" />
  <title>Page title</title>
</head>
<body>
  <header data-transition-layer="global-component">
    <!-- Navigation -->
  </header>

  <main>
    <!-- Page content -->
  </main>

  <footer data-transition-layer="global-component">
    <!-- Footer -->
  </footer>

  <!-- Scripts — order matters for dependencies -->
  <script src="./zazz/scripts/utils.js"></script>
  <script src="./zazz/scripts/reveal.js"></script>
  <script src="https://unpkg.com/embla-carousel/embla-carousel.umd.js"></script>
  <script src="./zazz/scripts/embla.js"></script>
  <script src="./zazz/scripts/navigation.js"></script>
</body>
</html>
```

The reset gives `<body>`, `<main>`, and the section landmarks their own layout and container context, so this skeleton is doing more than grouping markup — see [`_reset.css`](/docs/core-files/reset).

## Stylesheet load order

`main.css` handles this for you, but if you ever link files individually the order is fixed:

1. `_layers.css` — declares `@layer variables, reset, components, utilities;` ([why this comes first](/docs/core-files/layers))
2. `_properties.css` — registered `@property` declarations
3. `_variables.css` — global design tokens
4. `_reset.css` — element baselines
5. Component files — in any order
6. `_utilities.css` — atomic override classes (load last)

With HTTP/2–3 multiplexing and server compression, the single `main.css` link is almost always the better choice — there's nothing to keep in order.

## Script load order

Scripts declare dependencies through `<script>` tag order. Load them at the end of `<body>`:

1. `utils.js` — shared parsing utilities (load first)
2. `reveal.js` — standalone, no dependencies
3. Embla CDN bundles — UMD builds from unpkg/CDN
4. `embla.js` — requires `utils.js` and the Embla globals
5. `navigation.js` — standalone, app-level (skip it inside component preview iframes)

All scripts auto-initialize on `DOMContentLoaded`. Standard pages need no manual init calls. The conventions behind these files live in [Scripting](/docs/core-files/scripting).

## View transition layers

For page transitions to animate cleanly — cross-document, or SPA-style via `navigation.js` — mark your landmarks with `data-transition-layer`:

```html
<header data-transition-layer="global-component">...</header>
<main>...</main>
<footer data-transition-layer="global-component">...</footer>
```

These map to `view-transition-name` values that `_page-transitions.css` uses to animate the `<main>` content while the header and footer stay put during a navigation.
