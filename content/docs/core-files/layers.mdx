`layers.css` is the smallest file in Zazz and the most important. It declares the order of the [cascade layers](https://developer.mozilla.org/en-US/docs/Web/CSS/@layer) every other stylesheet writes into, and that order, not selector specificity, decides which rule wins.

```css
@layer legacy, reset, zazz, migrations;

@layer zazz {
  @layer variables, components, utilities;
}
```

That's the entire file. Because it has to establish the order before any layer has content, it loads first; `zazz.css` `@import`s it ahead of everything else.

## Why layers instead of specificity

In ordinary CSS, the more specific selector wins. That's why you end up with `.card .button.is-primary` and the occasional `!important`: you're escalating specificity to override something.

Cascade layers change the rule: **a later layer beats an earlier layer regardless of specificity.** A single-class utility in `utilities` overrides a multi-selector component rule in `components`, every time, with no specificity tricks.

```html
<!-- .button sets its own background in @layer zazz.components.
     .bg-primary sets one in @layer zazz.utilities — and wins, cleanly. -->
<button class="button bg-primary">Save</button>
```

This is the mechanism behind Zazz's [golden rule](/docs/core-concepts/utility-classes): compose with utilities and they always have the final say, so you rarely write custom CSS to override a component.

## The layer order

Six layers in all, reading from lowest priority to highest:

| Layer             | Holds                                                                                                                                       | File                                                                                                     |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| `legacy`          | Your existing (pre-Zazz) CSS, imported with `layer(legacy)`. Lowest priority, so Zazz overrides it wherever the two overlap.                | your app's entry stylesheet                                                                               |
| `reset`           | Native-element baselines and re-skinned controls                                                                                            | [`_reset.css`](/docs/core-files/reset)                                                                   |
| `zazz.variables`  | Design tokens, the `:root` custom properties everything reads                                                                               | [`_variables.css`](/docs/core-concepts/theme-variables) + each component's `@layer variables` block |
| `zazz.components` | The components themselves: `.button`, `.card`, `.input`…                                                                                     | each `_*.css` component file                                                                              |
| `zazz.utilities`  | Atomic, single-purpose classes that win last inside Zazz                                                                                     | [`_utilities.css`](/docs/core-concepts/utility-classes)                                                  |
| `migrations`      | Temporary shims that map old class names to Zazz tokens, plus the `data-legacy-css` reverts. On top, so it can override anything mid-migration. | [`_legacy.css`](/docs/getting-started/migrating)                                                          |

`@property` registrations (in `_properties.css`) sit outside the layers, since registration is layer-independent, but their tokens are consumed throughout.

Two parts of the order are worth calling out. **`reset` is its own top-level layer** below all of Zazz, so component and utility rules always outrank native-element baselines. And **`migrations` sits on top of `zazz.utilities`,** at the very top of the stack: a class-translation shim, or a `data-legacy-css` revert, has to win over even a utility class for old markup to render correctly before you rewrite it. See [Migrating to Zazz](/docs/getting-started/migrating) for how that plays out.

## Unlayered styles beat everything

There's one level above all six layers: **unlayered CSS.** Any rule you write outside a `@layer` block overrides every layered rule, no matter how specific the layered selector is.

```css
/* No @layer wrapper — this beats Zazz's components, utilities, and migrations. */
.button {
  border-radius: 0;
}
```

That makes unlayered CSS the ultimate escape hatch and an easy footgun. Prefer overriding a token, or adding your rule to the right layer, so you stay inside the system. See [Adding custom styles](/docs/core-concepts/custom-styles). Reach for unlayered CSS only when you deliberately want to trump everything.

## Beating Zazz utilities

Earlier versions of Zazz shipped a separate `overrides` layer for app rules that needed to beat utilities. That layer is gone. When overriding a token isn't enough and an app rule has to win over a utility, declare your own layer **after** Zazz. The first `@layer` statement sets the order, so a name added at the end outranks everything Zazz ships:

```css
@layer legacy, reset, zazz, migrations, app;

@layer app {
  /* beats Zazz utilities, without the footgun of unlayered CSS */
  .button {
    border-radius: 0;
  }
}
```

This keeps your overrides layered and predictable. Reach for it only when overriding a [token](/docs/core-concepts/theme-variables) won't do.

## Scoped isolation with `_legacy.css`

`_legacy.css` is an optional file (disabled by default in `zazz.css`) for mixed codebases. It carries the `data-legacy-css` attribute hooks and is the home for your migration shims. Uncomment its import when you're migrating an existing project alongside Zazz:

```css
/* zazz.css — uncomment to enable */
@import "./_legacy.css";
```

Once enabled, two `data-legacy-css` attributes control which layer applies to a subtree:

| Attribute                | Effect                                                                                                                                    |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `data-legacy-css="none"` | Strips the `legacy` layer from this element and its children. Zazz applies; your old CSS does not. Use on sections you've converted.      |
| `data-legacy-css="only"` | Strips the `zazz` layer from this element and its children. Your old CSS applies; Zazz does not. Use on sections you haven't touched yet. |

Both use `:where()` for zero specificity and `all: revert-layer` rather than `all: revert`, so they're surgical: they remove one layer without disturbing the others. A `!none` / `!only` pair does the same with `all: revert !important` from the `migrations` layer when you need to force the boundary.

```html
<!-- This section is converted — block legacy styles, keep Zazz -->
<section data-legacy-css="none">
  <button class="button" data-variant="primary">Save</button>
</section>

<!-- This widget is untouched — block Zazz, keep legacy -->
<div data-legacy-css="only">
  <div class="legacy-date-picker">...</div>
</div>
```

For a full migration walkthrough, see [Migrating to Zazz](/docs/getting-started/migrating).

## Browser support

Cascade layers are [Baseline](https://web.dev/baseline) and widely available, supported since 2022 across Chrome, Edge, Firefox, and Safari. No fallback is required.
