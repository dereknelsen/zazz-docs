`_layers.css` is the smallest file in Zazz and the most important. It declares the order of the [cascade layers](https://developer.mozilla.org/en-US/docs/Web/CSS/@layer) every other stylesheet writes into, and that order, not selector specificity, decides which rule wins.

```css
@layer variables, reset, legacy, zazz, migrations;

@layer zazz {
  @layer components, utilities;
}
```

That's the entire file. Because it has to establish the order before any layer has content, it loads first; `main.css` `@import`s it ahead of everything else.

## Why layers instead of specificity

In ordinary CSS, the more specific selector wins. That's why you end up with `.card .button.is-primary` and the occasional `!important`: you're escalating specificity to override something.

Cascade layers change the rule: **a later layer beats an earlier layer regardless of specificity.** A single-class utility in `utilities` overrides a multi-selector component rule in `components`, every time, with no specificity tricks.

```html
<!-- .button sets its own background in @layer zazz.components.
     .bg-primary sets one in @layer zazz.utilities — and wins, cleanly. -->
<button class="button bg-primary">Save</button>
```

This is the mechanism behind Zazz's [golden rule](/docs/core-concepts/utility-classes): compose with utilities and they always have the final say, so you rarely write custom CSS to override a component.

## The layer order

Five top-level layers, reading from lowest priority to highest:

| Layer             | Holds                                                                                         | File                                                                                          |
| ----------------- | --------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| `variables`       | Design tokens, the `:root` custom properties everything reads                                 | [`_variables.css`](/docs/base/variables) + each component's `@layer variables` block |
| `reset`           | Native-element baselines and re-skinned controls                                              | [`_reset.css`](/docs/base/reset)                                                         |
| `legacy`          | Your existing (pre-Zazz) CSS, imported with `layer(legacy)`. Zazz overrides it wherever the two overlap. | your app's entry stylesheet                                                          |
| `zazz`            | Everything Zazz ships, as two sublayers: `components` then `utilities`                         | `ui/_*.css` (components) and [`_utilities.css`](/docs/core-concepts/utility-classes)           |
| `migrations`      | Temporary shims for old markup during migration. On top, so it can override anything mid-rewrite. | optional `migrations.css` you add                                                          |

Inside `zazz`, `utilities` comes after `components`, so a utility class always beats a component rule. The full priority order, lowest to highest, is `variables`, `reset`, `legacy`, `zazz.components`, `zazz.utilities`, `migrations`.

Two parts of the order are worth calling out. **`variables` and `reset` are their own top-level layers, below the rest of Zazz**, so component and utility rules always outrank both design-token defaults and native-element baselines. And **`legacy` sits above them but below `zazz`,** so the framework wins over your pre-Zazz CSS wherever they overlap, while `migrations` sits at the very top so a translation shim can win over even a utility class for old markup that hasn't been rewritten yet.

`@property` registrations live inline in [`_variables.css`](/docs/base/variables). Registration is layer-independent, but the tokens it types are consumed throughout.

## Unlayered styles beat everything

There's one level above all five layers: **unlayered CSS.** Any rule you write outside a `@layer` block overrides every layered rule, no matter how specific the layered selector is.

```css
/* No @layer wrapper — this beats Zazz's components, utilities, and migrations. */
.button {
  border-radius: 0;
}
```

That makes unlayered CSS the ultimate escape hatch and an easy footgun. Prefer overriding a token, or adding your rule to the right layer, so you stay inside the system. See [Adding custom styles](/docs/core-concepts/custom-styles). Reach for unlayered CSS only when you deliberately want to trump everything.

## Beating Zazz utilities

When overriding a token isn't enough and an app rule has to win over a utility, declare your own layer **after** Zazz instead of dropping to unlayered CSS. The first `@layer` statement sets the order, so a name added at the end outranks everything Zazz ships:

```css
@layer variables, reset, legacy, zazz, migrations, app;

@layer app {
  /* beats Zazz utilities, without the footgun of unlayered CSS */
  .button {
    border-radius: 0;
  }
}
```

This keeps your overrides layered and predictable. Reach for it only when overriding a [token](/docs/base/variables) won't do.

## Bringing along legacy CSS

Migrating an existing project? Import your old stylesheet into the `legacy` layer. `main.css` leaves a commented slot for exactly this:

```css
/* main.css — uncomment and point at your stylesheet */
@import "./your-legacy.css" layer(legacy);
```

Because `legacy` sits below `zazz`, Zazz wins wherever the two overlap, and your untouched markup keeps rendering from its old rules until you convert it. For shims that have to win _over_ Zazz during the rewrite — mapping an old class name onto Zazz tokens, say — add a `migrations.css` at the top of the stack (there's a commented `@import` slot for it at the bottom of `main.css`); the `migrations` layer outranks the utilities. For surgical, per-region isolation you can also reach for [`@scope`](https://developer.mozilla.org/en-US/docs/Web/CSS/@scope) donut scoping. See [Migrating to Zazz](/docs/getting-started/migrating) for the full walkthrough.

## Browser support

Cascade layers are [Baseline](https://web.dev/baseline) and widely available, supported since 2022 across Chrome, Edge, Firefox, and Safari. No fallback is required.
