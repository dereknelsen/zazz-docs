title: Input
description: Single-line text entry. One class for every text-like type, plus input-group composition for icons and addons.
---

A text input is a single-line field for free-form entry. Zazz styles the native `<input>` element with a single `.input` class that covers every text-like type â€” `text`, `email`, `tel`, `url`, `search`, `password`, `number`, `date`, and more. The `type` only swaps the keyboard or picker; the box is identical. Inputs and buttons share the same height (`--field-height`, which resolves to `--step-8`) and corner radius (`--field-radius`, which resolves to `--radius-md`) so they line up cleanly in forms.

## Default

<ComponentPreview name="input-default" />

```html
<input class="input" type="text" placeholder="Placeholder" />
```

| Token      | Value                                                                 |
| ---------- | --------------------------------------------------------------------- |
| Width      | `100%`                                                                |
| Height     | `--field-height` (`--step-8`)                                         |
| Padding    | `0 --field-padding` (`--step-2_5`)                                    |
| Border     | `1px solid var(--field-border)` (defaults to `--border`)              |
| Radius     | `--field-radius` (defaults to `--radius-md`)                          |
| Background | `--field-background` (`--input`)                                      |
| Color      | `--field-foreground` (`--input-foreground`)                           |
| Font       | `--font-size-sm`                                                      |
| Hover      | Border color shifts to `--field-border--hover`                        |
| Focus      | Border `--primary`, plus the [focus ring](/docs/reference/focus-ring) |

## With a leading icon

To attach an icon, wrap the control in an `.input-group` â€” a single bordered shell that fuses the input with addons. The shell is authored as a `<label>` so clicking a non-interactive addon focuses the control, and `:focus-within` lifts the whole field while the nested input goes borderless and transparent. Addons come **after** the control in the DOM; `data-align` positions them. The default alignment, `inline-start`, places the addon visually before the control.

<ComponentPreview name="input-icon-leading" />

```html
<label class="input-group">
  <input class="input" type="text" placeholder="Placeholder" />
  <span class="input-group__addon">
    <svg><!-- menu icon --></svg>
  </span>
</label>
```

## With a trailing icon

Set `data-align="inline-end"` on the addon to place it after the control.

<ComponentPreview name="input-icon-trailing" />

```html
<label class="input-group">
  <input class="input" type="search" placeholder="Searchâ€¦" aria-label="Search" />
  <span class="input-group__addon" data-align="inline-end">
    <svg><!-- search icon --></svg>
  </span>
</label>
```

## Anatomy

| Part  | Class                | Notes                                                                                                |
| ----- | -------------------- | ---------------------------------------------------------------------------------------------------- |
| Shell | `input-group`        | Optional, required only when adding addons. A `<label>` whose `:focus-within` lifts the whole field. |
| Input | `input`              | The `<input>` element. Inside a group it sheds its own border and background.                        |
| Addon | `input-group__addon` | Holds an `<svg>`, a `.button`, `<kbd>`, or `.input-group__text`. `data-align` positions it.          |
| Text  | `input-group__text`  | Inline text for units, protocols, or handles (`https://`, `USD`, `@user`).                           |

Addon alignment via `data-align`: `inline-start` (default), `inline-end`, `block-start`, and `block-end`. The `block-*` values span a full-width row â€” that's how a textarea grows a toolbar above or below it.

## Form composition

Wrap each input in a `.field` with a `.field__label`. The `.field` is a grid that stacks the label (row 1), the control (row 2), and a `.field__description` (row 3). Inside `.field__description`, add `.field__hint` for muted helper copy and `.field__error` for the destructive-themed validation message.

```html
<div class="field">
  <label class="field__label" for="email">Email</label>
  <input
    class="input"
    id="email"
    type="email"
    required
    placeholder="you@example.com"
    aria-describedby="email-hint"
  />
  <div class="field__description">
    <span class="field__hint" id="email-hint">We'll only use this to send receipts.</span>
    <span class="field__error" role="alert">Please enter a valid email address.</span>
  </div>
</div>
```

## States

- **Default.** Border `--border`, background `--input`.
- **Hover.** Border lifts to `--field-border--hover` to signal interactivity.
- **Focus.** Border switches to `--primary`; the focus ring is a `:focus-visible` **outline** (`outline-color: var(--ring)`), offset by `--ring-offset`. Modern outlines follow the border radius, so the ring hugs the field's rounded corners.
- **Disabled.** Set `disabled` on the input. The shared disabled state applies `opacity: 0.5` and `cursor: not-allowed`.
- **Invalid.** Validation hangs off `:user-invalid`, so the border, ring, and label only turn `--destructive` â€” and the `.field__error` reveals while the `.field__hint` hides â€” _after_ the user commits a value (blur or submit), never while typing.

## Tokens summary

| Property         | Token                                                               |
| ---------------- | ------------------------------------------------------------------- |
| Height           | `--field-height` (`--step-8`)                                       |
| Padding (inline) | `--field-padding` (`--step-2_5`)                                    |
| Border           | `--field-border` (`--border`), `--primary` (focus)                  |
| Radius           | `--field-radius` (`--radius-md`)                                    |
| Background       | `--field-background` (`--input`)                                    |
| Foreground       | `--field-foreground` (`--input-foreground`)                         |
| Icon size        | `--field-icon-size` (`--step-4`)                                    |
| Focus ring       | `--ring` outline (`--ring-size` / `--ring-style` / `--ring-offset`) |

To retune the radius for the whole form family, override `--field-radius` (declared in `_fields.css`), or shift everything with `--radius-multiplier`.

## Accessibility

- Every input needs a `<label class="field__label">` that's either visible or screen-reader-only via [`sr-only`](/docs/reference/screen-readers).
- The `placeholder` is not a label substitute. Use it for examples or hints; the label still needs to be there.
- For icon-only signaling (search icon, success indicator), the icon should be decorative (`aria-hidden="true"`). The input itself carries the accessible name from its label.
- Required fields should use `required` and a visible cue in the label (an asterisk or "required" text).
