---
title: Migrating to Zazz
description: Adopt Zazz incrementally on top of an existing CSS codebase, using cascade layers, data-legacy-css attributes, and optional @scope donut scoping so the two systems never fight while you migrate.
---

You don't have to rewrite your CSS to start using Zazz. Because Zazz lives entirely in cascade
layers and styles at zero specificity, you can drop it into an existing project and let it take
over screen by screen. No big-bang rewrite, no `!important` war with your old styles.

## The core problem: unlayered styles beat layered ones

Zazz puts every rule inside a layer (`legacy`, `zazz.*`, or `overrides`). The cascade has a
rule that surprises people: **any unlayered author rule beats every layered rule, regardless of
specificity.** Your existing CSS is almost certainly unlayered, so on its own it would override
Zazz's components and utilities everywhere the two overlap.

The fix is to give your old CSS a layer of its own, ordered _below_ Zazz. From there, everything
else follows.

## Step 1: Layer your old CSS below Zazz

In the entry stylesheet you load before Zazz, declare the full layer order up front, then import
your legacy CSS into the lowest layer:

```css
/* entry.css - loaded before Zazz */
@layer legacy, zazz, overrides;

@import "./legacy/app.css" layer(legacy); /* old CSS → lowest priority */
@import "./zazz/styles/main.css"; /* includes _legacy.css + _migrations.css */
```

The **first** `@layer` statement wins for ordering, so declaring it here (before `main.css`
re-declares its own subset) locks `legacy` underneath everything Zazz ships. Now where both
systems style the same thing, Zazz wins; everywhere Zazz is silent, your old CSS still applies. On
day one, nothing visually breaks.

<Callout>
  `zazz.migrations` sits above Zazz's `reset` and `components` sublayers but below `zazz.utilities`.
  That ordering matters in Step 5: a translated class like `.btn-primary` must outrank the element
  baseline in `reset`, while Zazz utilities (`rounded-full`, `bg-primary`) must still override it.
</Callout>

## Step 2: Isolate regions with `data-legacy-css`

`_legacy.css` (disabled by default) adds two HTML attributes that let you opt any subtree into
one system or the other without writing any CSS. Uncomment its import in `main.css`:

```css
/* main.css */
@import "./_legacy.css"; /* remove the comment to enable */
```

Then use the attributes on any element:

| Attribute                | Effect                                                                                             |
| ------------------------ | -------------------------------------------------------------------------------------------------- |
| `data-legacy-css="none"` | Strips the `legacy` layer from this element and its children. Zazz applies; your old CSS does not. |
| `data-legacy-css="only"` | Strips the `zazz` layer from this element and its children. Your old CSS applies; Zazz does not.   |

Both use `:where()` for zero specificity and `all: revert-layer` rather than `all: revert`, so
they're surgical: they remove one layer without disturbing the others or the browser's defaults.

```html
<!-- This section is converted — stop legacy CSS from reaching in -->
<section data-legacy-css="none">
  <button class="button" data-variant="primary">Save</button>
</section>

<!-- This widget is untouched — stop Zazz from reaching in -->
<div data-legacy-css="only">
  <div class="legacy-date-picker">...</div>
</div>
```

For most incremental migrations, this is all you need. If your legacy and Zazz regions are
more deeply interleaved, `@scope` gives you finer control.

## Step 3: Bound regions with `@scope` (for complex cases)

`@scope` applies a block of rules to a region and, with a lower boundary, stops them at a subtree:
"**donut scoping**." The donut hole is the part you leave alone:

```css
@scope (.app) to (.legacy-widget) {
  /* matches inside .app, but NOT inside any .legacy-widget */
}
```

This is useful when you need migration rules (not just layer isolation) to stop at a boundary— for
example, `_migrations.css` shims that should apply to a converted section but not reach into a
nested legacy widget inside it. Unlike a `.app :not(.legacy-widget *)` selector, `@scope` accounts
for hierarchical proximity, so it still behaves correctly when regions nest inside each other.

## Step 4: Blank-slate a region with `all: revert`

Once Zazz is on the page, its `reset` sublayer restyles native elements globally. That's what you
want for new UI, but it also reaches into old markup. To hand a region fully to Zazz, reset it to
the browser's defaults first with `all: revert`, which drops **every** property back to the
user-agent value, erasing both your legacy styles and Zazz's on that element:

```css
@scope (.takeover) to (.legacy-widget) {
  :scope,
  :scope * {
    all: revert; /* ignore all author CSS; start from UA defaults */
  }
}
```

Bounding the reset inside `@scope` is what lets you avoid a global `* { all: revert }`, which the
cascade can't cleanly override and which web components and lower layers can't escape.

<Callout>
  `all: revert` is the sledgehammer. It removes Zazz too. Its surgical sibling, `all: revert-layer`,
  drops only to the **previous cascade layer**. Put a `revert-layer` reset inside the `legacy` layer
  and it strips your old styles while Zazz's higher layers keep applying on top. Reach for
  `revert-layer` when you mean "remove my old styles but keep Zazz," and `revert` when you want a
  true blank slate (for example, to quarantine a third-party widget from both systems).
</Callout>

## Step 5: Translate old class names in `_migrations.css`

The last piece bridges markup you haven't rewritten yet. `_migrations.css` re-points your old
class names at Zazz's component tokens, so existing HTML renders as Zazz **before** you touch it.
Because Zazz components are token-driven, the shim is small and stays themeable, so light/dark and
brand changes flow through for free.

Here `.btn-primary` is translated to the Zazz button semantic tokens. Old modifier classes map to
Zazz variants the same way `data-variant` does, by reassigning the component tokens, not restating
the component rule:

```css
/* _migrations.css - a temporary bridge. Delete each rule as you rewrite the markup. */

@layer zazz.migrations {
  @scope (.app) to (.legacy-widget) {
    .btn-primary {
      --button-background: var(--primary);
      --button-foreground: var(--primary-foreground);
    }
  }
}
```

The same move works for any component: a legacy `.card` can adopt card tokens, a `.tag` can adopt
badge tokens, and so on. Each translated rule is a stepping stone. Once a screen's markup is
updated to native Zazz classes and attributes, delete its shim. When `_migrations.css` is empty,
the migration is done.

## A migration path

1. **Layer.** Add the `@layer` order and import your old CSS into `legacy` (Step 1). Ship it.
   Nothing should change yet.
2. **Isolate.** Enable `_legacy.css` and add `data-legacy-css` attributes to wall off converted
   and unconverted sections (Step 2). No CSS to write.
3. **Scope (if needed).** For deeply interleaved regions, use `@scope` donut scoping to bound
   migration rules more precisely (Step 3).
4. **Reset where needed.** Blank-slate stubborn regions with `all: revert` / `revert-layer` so
   legacy styling stops leaking in (Step 4).
5. **Translate, then replace.** Add `_migrations.css` rules to make old markup look right now
   (Step 5), then rewrite the HTML to real Zazz classes and delete the shim, screen by screen.

## Browser support

<Callout type="warn">
  `@layer`, `all: revert`, and `revert-layer` are **Baseline widely available**, safe everywhere
  with no fallback. The `data-legacy-css` attributes are pure CSS built on those same features, so
  they share the same support floor. `@scope` is **Baseline 2024 (newly available)**: Chrome/Edge
  118, Safari 17.4, Firefox 128. If you must support older browsers, skip the `@scope` step and lean
  on `@layer` plus the `data-legacy-css` attributes instead.
</Callout>
