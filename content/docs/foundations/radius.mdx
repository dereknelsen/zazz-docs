**Most radius systems are a scale. Zazz adds a per-component layer so nested corners look right.**

Radius works in two layers plus a per-component alias: a global multiplier that scales everything at once, a short semantic scale (`none`, `xs` through `xl`, `full`), and a local prop each component declares for itself that points at a step on the scale.

The semantic layer covers most direct decisions. The local-prop layer solves the nesting problem (buttons inside cards, badges on buttons) without designers having to do the math each time, and lets you retune one component without disturbing the rest.

## Semantic radii

The base scale. Five named radii plus `none` and `full`.

```css
--radius-none: 0rem;
--radius-xs: calc(var(--step-1) * var(--radius-multiplier)); /* ≈ 4px  */
--radius-sm: calc(var(--step-1_5) * var(--radius-multiplier)); /* ≈ 6px  */
--radius-md: calc(var(--step-2_5) * var(--radius-multiplier)); /* ≈ 10px */
--radius-lg: calc(var(--step-4) * var(--radius-multiplier)); /* ≈ 16px */
--radius-xl: calc(var(--step-7) * var(--radius-multiplier)); /* ≈ 28px */
--radius-full: 9999rem;
```

Each numbered radius derives from a step on the spacing scale, multiplied by `--radius-multiplier`. At the settled 4px interval the scale lands at roughly `4 / 6 / 10 / 16 / 28px`. The scale is intentionally short. Five rungs cover most surfaces, and `--radius-full` handles capsules and pills.

`--radius-none` exists so a component can switch off its rounding without resorting to a literal `0`. Tokens stay tokens; nothing breaks the convention.

## Per-component radii

Components don't reach into the scale at the point of use. Each one declares a local radius prop in its own partial and points it at a semantic step. The component then consumes its own prop.

```css
/* _button.css */
.button {
  --button-radius: var(--radius-md); /* ≈ 10px */
  border-radius: var(--button-radius);
}

/* _badge.css */
.badge {
  --badge-radius: var(--radius-sm); /* ≈ 6px — a gently rounded chip */
  border-radius: var(--badge-radius);
}

/* _fields.css — form controls share one field radius */
.field {
  --field-radius: var(--radius-md); /* ≈ 10px */
}
```

Not every component needs a local prop. `_dialog.css` reaches for `--radius-lg` directly, because nothing nested inside it needs to retune the dialog's corners independently.

The reason for the local prop: nested radii look broken when the inner element's radius is too close to the container's. A button inside a card with the same radius reads as not-quite-aligned to the eye. By giving the button (`--radius-md`) and the card surface (`--radius-lg`) different semantic steps, the math comes out right by default.

The local prop also lets you retune one component without disturbing the rest. Make every button a capsule:

```css
:root {
  --button-radius: var(--radius-full);
}
```

Field rounding, badge shapes, and dialog corners stay where they are.

## Global multiplier

`--radius-multiplier` is the lever for sharpening or rounding every radius at once.

```css
--radius-multiplier: 1;
```

Default is `1`. Bump it for rounder corners across the product; drop it for a sharper, more architectural feel. Components don't care. They consume their local prop, which resolves through the scale, which resolves through the multiplier.

```css
.architectural {
  --radius-multiplier: 0; /* sharp corners everywhere */
}

.friendly {
  --radius-multiplier: 1.5; /* rounder, softer */
}
```

`--radius-md` at multiplier `1.5` becomes `step-2_5 × 1.5`. The proportions between `xs`, `sm`, `md`, `lg`, and `xl` stay constant regardless of multiplier. They all scale together.

`--radius-none` and `--radius-full` are unaffected by the multiplier. `none` is always `0`. `full` is always capsule.

## Utilities

Apply a scale step directly with a rounding utility:

```html
<div class="rounded-md">…</div>
```

`rounded-0`, `rounded-xs`, `rounded-sm`, `rounded-md`, `rounded-lg`, `rounded-xl`, and `rounded-full` map onto the matching scale step (with `rounded-0` setting a literal `0`).

## Nested radius math

The classic problem: a button (≈10px radius) sits inside a card (≈16px radius). If the button's radius is too close to the card's, the visual rhythm breaks. Zazz's per-component defaults ship with the math worked out:

| Component | Local prop / source    | Default       | Resolves to |
| --------- | ---------------------- | ------------- | ----------- |
| Dialog    | `--radius-lg` (direct) | `--radius-lg` | ≈ 16px      |
| Button    | `--button-radius`      | `--radius-md` | ≈ 10px      |
| Field     | `--field-radius`       | `--radius-md` | ≈ 10px      |
| Badge     | `--badge-radius`       | `--radius-sm` | ≈ 6px       |

A surface at 16px with a button inside at 10px reads cleanly. The roughly 2:3 ratio gives the eye enough difference. The badge is a gently rounded chip at `--radius-sm`, not a capsule, so it nests against a button or input without fighting the surrounding corners.

The defaults compose well. When you retune a component for a project, check the most-common nesting paths in your UI and verify the inner radius stays _visibly_ smaller than the outer.

<Accordions>
  <Accordion title="What's the rule for nested-radius math?">
    The strict geometric rule: the inner radius should equal the outer radius minus the inner element's distance from the outer edge. If a card has a 16px radius and 16px of internal padding, an element flush against the card's inner edge would need a 0px radius for the curves to be concentric.

    Most of the time you don't want elements pressed flush. They sit on a stack with some gap, so the math softens. A good rule of thumb: the inner radius should be roughly half the outer, and never larger than two-thirds.

    Zazz's per-component defaults follow this:

    - Dialog (`--radius-lg`, ≈16px) → button (`--button-radius` → `--radius-md`, ≈10px). Under two-thirds.
    - Button (`--button-radius` → `--radius-md`, ≈10px) → badge (`--badge-radius` → `--radius-sm`, ≈6px). Just over half.
    - Field (`--field-radius` → `--radius-md`, ≈10px) shares the button's step, so controls and buttons sit on the same rhythm.

    When you retune a component, pick a step that preserves this kind of ratio. A button at `--radius-lg` inside a card at `--radius-md` looks visibly wrong; a button at `--radius-md` inside a card at `--radius-xl` reads clean.
